# Процесс хотфиксов WakeLink

> Экстренные исправления для критических проблем в WakeLink v1.x

## 🚨 Когда использовать этот процесс

Используйте процесс хотфикса для:
- **S1 Критический**: Система недоступна, нарушение безопасности, потеря данных
- **S2 Высокий**: Основная функция не работает, обходного пути нет

Для проблем S3/S4 используйте стандартный цикл выпуска.

## 🔧 Пошаговое руководство

### 1. Создание ветки хотфикса

```bash
# Убедитесь, что main актуален
git checkout main
git pull origin main

# Создайте ветку хотфикса
git checkout -b hotfix/issue-{NUMBER}-{short-desc}

# Пример:
git checkout -b hotfix/issue-142-auth-bypass
```

### 2. Реализация исправления

**Правила:**
- Исправляйте ТОЛЬКО заявленную проблему
- Никакого рефакторинга или попутных изменений
- Добавьте регрессионный тест для бага
- Обновите CHANGELOG.md

```bash
# Make changes
git add -A
git commit -m "fix: [S1] auth bypass in session validation

Fixes #142

- Added missing token expiry check
- Added regression test
"
```

### 3. Ускоренное ревью

Для S1 (Критический):
- Напишите ревьюеру напрямую (Slack/Discord DM)
- Асинхронное одобрение допустимо: ревьюер может одобрить через комментарий
- Достаточно одного одобрения

Для S2 (Высокий):
- Стандартный PR, но ускоренный
- SLA на ревью — 2 часа

### 4. Чек-лист дымового теста

```markdown
- [ ] Сервер запускается без ошибок
- [ ] Эндпоинты здоровья отвечают (GET /health)
- [ ] Аутентификация работает (процесс входа)
- [ ] Основная функция работает (команда пробуждения отправляется)
- [ ] Нет новых ошибок в логах (наблюдение 30 сек)
```

### 5. Деплой

```bash
# SSH to production server
ssh deploy@wakelink-prod

# Pull and restart
cd /opt/wakelink
docker-compose -f docker-compose.prod.yml pull api
docker-compose -f docker-compose.prod.yml up -d api

# Verify
docker-compose logs -f api --tail=50
```

### 6. Мониторинг после деплоя

**Наблюдайте 30 минут:**
- Частота ошибок в Grafana: должна вернуться к базовому уровню
- Задержка: не должна увеличиваться
- Нет новых исключений в логах

**Метрики для проверки:**
- `http_requests_total{status="5xx"}` — должно быть близко к нулю
- `http_request_duration_seconds{quantile="0.99"}` — менее 500 мс
- `wakelink_active_websocket_connections` — стабильно или растёт

### 7. Слияние и тег

```bash
# Merge hotfix to main
git checkout main
git merge --no-ff hotfix/issue-{NUMBER}-{short-desc}

# Tag release
git tag -a v1.0.X -m "Hotfix: {short description}"
git push origin main --tags

# Cherry-pick to develop (if applicable)
git checkout develop
git cherry-pick -x <commit-hash>
git push origin develop

# Delete hotfix branch
git branch -d hotfix/issue-{NUMBER}-{short-desc}
git push origin --delete hotfix/issue-{NUMBER}-{short-desc}
```

## 🔄 Процедура отката

Если хотфикс вызвал проблемы:

### Немедленный откат (< 5 минут)

```bash
# SSH to production
ssh deploy@wakelink-prod

# Rollback to previous image
cd /opt/wakelink
docker-compose -f docker-compose.prod.yml stop api
docker-compose -f docker-compose.prod.yml up -d api --no-recreate

# Or specify exact previous tag
docker-compose -f docker-compose.prod.yml pull api:v1.0.{X-1}
docker-compose -f docker-compose.prod.yml up -d api
```

### Полный откат

```bash
# Revert the hotfix commit on main
git checkout main
git revert HEAD
git push origin main

# Re-deploy
# Follow normal deploy process
```
