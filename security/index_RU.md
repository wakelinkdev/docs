# Обзор безопасности

WakeLink разработан с учётом безопасности как основного принципа.

## Модель безопасности

```
┌────────────────────────────────────────────────────────────┐
│                    Security Layers                          │
├────────────────────────────────────────────────────────────┤
│  ┌──────────────────────────────────────────────────────┐  │
│  │         Transport Security (TLS 1.3)                 │  │
│  │    • Server authentication (certificates)            │  │
│  │    • Perfect forward secrecy                         │  │
│  └──────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────┐  │
│  │         Application Encryption (XChaCha20)           │  │
│  │    • End-to-end encryption                           │  │
│  │    • Server cannot read commands                     │  │
│  └──────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────┐  │
│  │         Message Integrity (HMAC-SHA256)              │  │
│  │    • Packet authenticity                             │  │
│  │    • Tamper detection                                │  │
│  └──────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────┐  │
│  │         Replay Protection (Chain Hashing)            │  │
│  │    • Sequence numbers                                │  │
│  │    • Packet chaining                                 │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
```

## Ключевые возможности

| Функция | Реализация | Назначение |
|---------|------------|------------|
| **Сквозное шифрование** | XChaCha20-Poly1305 | Сервер не может читать команды |
| **Аутентификация** | JWT + токены устройств | Подтверждение личности |
| **Целостность** | HMAC-SHA256 | Обнаружение подделки |
| **Защита от воспроизведения** | Последовательность + цепочка | Предотвращение атак повтора |
| **Безопасность транспорта** | TLS 1.3 | Защита при передаче |
| **Формирование ключей** | HKDF-SHA256 | Безопасное расширение ключей |

## Модель угроз

### От чего защищает WakeLink

✅ **Прослушивание** — команды зашифрованы сквозным шифрованием
✅ **Атака «человек посередине»** — TLS + проверка сертификатов
✅ **Атаки воспроизведения** — порядковые номера + хеш-цепочка
✅ **Внедрение команд** — строгая валидация команд
✅ **Несанкционированное включение** — токен-аутентификация
✅ **Компрометация сервера** — сквозное шифрование (сервер не читает нагрузку)

### От чего WakeLink НЕ защищает

❌ **Физический доступ к ESP-устройству** — злоумышленник может извлечь ключи
❌ **Скомпрометированное клиентское устройство** — вредоносное ПО может похитить токены
❌ **DDoS-атаки** — смягчаются, но не предотвращаются полностью
❌ **Физическое принуждение** — метод «резинового шланга»

## Безопасность протокола (EWSP v1.0)

### Шифрование

- **Алгоритм**: XChaCha20-Poly1305 (AEAD)
- **Nonce**: 192-битный, уникальный для каждого сообщения
- **Размер ключа**: 256 бит

XChaCha20-Poly1305 обеспечивает:
- Конфиденциальность (шифрование)
- Целостность (MAC)
- Аутентичность (AEAD)

### Формирование ключей

```
Pre-Shared Key (32 bytes)
        │
        ▼
    HKDF-SHA256
        │
    ┌───┴───┐
    ▼       ▼
Encryption  MAC
   Key      Key
(32 bytes) (32 bytes)
```

### Защита от воспроизведения

Два механизма работают совместно:

1. **Порядковые номера**
   - Монотонно возрастающие
   - Не сбрасываются в течение сессии
   - Отклоняются, если не превышают последнее зафиксированное значение

2. **Хеш-цепочка**
   - Каждый пакет включает хеш предыдущего пакета
   - Создаёт криптографическую цепочку
   - Прерывается при изменении или повторении любого пакета

```
Packet 1: prev=NULL, hash=SHA256(packet1)
Packet 2: prev=hash1, hash=SHA256(packet2)
Packet 3: prev=hash2, hash=SHA256(packet3)
```

## Аутентификация

### Аутентификация пользователей

- **Метод**: Email + пароль → JWT
- **Хранение паролей**: Argon2id (устойчивый к памяти)
- **Срок действия токена**: 24 часа (настраивается)
- **Обновление**: автоматическое при наличии действующего токена

### Аутентификация устройств

- **Метод**: предварительно распределённый токен устройства
- **Формат токена**: 32 байта случайных данных в шестнадцатеричной кодировке
- **Хранение**: зашифровано во флэш-памяти устройства
- **Перегенерация**: доступна через API

## Безопасность данных

### Данные в покое

| Данные | Защита |
|--------|--------|
| Пароли | Хеширование Argon2id |
| Токены устройств | Зашифрованы в БД |
| Сессионные ключи | Только в памяти, не сохраняются |
| Логи | Конфиденциальные данные не записываются |

### Данные в транзите

| Канал | Защита |
|-------|--------|
| API-вызовы | TLS 1.3 |
| WebSocket | WSS (TLS) |
| Команды | TLS + XChaCha20 |

## Сетевая безопасность

### Требуемые порты

| Порт | Направление | Назначение |
|------|-------------|------------|
| 443 | Исходящий | HTTPS/WSS к серверу |

Входящие порты в домашней сети не требуются.

### Рекомендации по брандмауэру

```bash
# Разрешить только исходящий HTTPS
iptables -A OUTPUT -p tcp --dport 443 -j ACCEPT
```

### Безопасность DNS

- Используйте DNS-over-HTTPS или DNS-over-TLS
- Строго проверяйте сертификаты
- Закрепите сертификаты для развёртываний с повышенными требованиями
